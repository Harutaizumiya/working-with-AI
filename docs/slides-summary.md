# Working with AI 内容总结

这份 slides 讨论的不是一套固定方法论，而是一套仍在快速变化中的个人实践：AI 的能力演进太快，今天有效的经验很可能很快过时，所以更重要的是理解框架、建立判断力，并持续调整自己和 AI 的协作方式。

## 1. 从 Tab 到 Agent

开篇先用一个很清楚的演进路径说明 AI 编程工具的变化：

- 最早是 `Tab` 式补全，AI 负责补一行代码。
- 后来变成 `Tab Tab`，可以同时处理多个插入点。
- 现在进入 `Agent` 阶段，用户不再告诉 AI “写哪一行”，而是告诉它“要去哪里”，AI 自己拆解、执行并验证。

紧接着，slides 用一页解释 `Prompt`、`Context`、`Agent` 三者的分工：

- `Prompt` 决定方向，告诉 AI 目标、约束和输出要求。
- `Context` 决定判断质量，告诉 AI 该基于什么信息做决定。
- `Agent` 决定事情能否真正做完，它会持续 plan、act、check、continue。

这个部分的核心观点是：同一个模型的表现，往往不是“模型能力”单独决定的，而是由 `Prompt + Context + Agent` 的组合同步塑形。

## 2. AI 编程正在进入更高自动驾驶等级

slides 把 AI 编程能力类比成“自动驾驶等级”：

- `L0` 是完全人工。
- `L1` 是行内补全。
- `L2` 是片段生成和 IDE Chat。
- `L3` 是 Coding Agent，人类 review。
- `L4` 和 `L5` 则进一步走向 Agent + AI Review 甚至 AI 工程团队。

这里的判断很明确：从 `L3` 开始，真正的瓶颈不再是“AI 会不会写”，而是“人是否足够信任这段代码”，所以 review 成为核心约束。

## 3. 当前的工具与方法选择

作者给出了一套比较明确的个人工具分工：

- `Codex` 负责写代码。
- `Gemini` 偏创意类工作。
- `ChatGPT` 偏 research。
- `Playwright / MCP` 这类能力作为工具接口补足环境交互。

随后用两页解释 `Skills` 和 `MCP`：

### Skills 是什么

`Skill` 被定义成给 agent 的“本地说明书”：

- 用自然语言描述某类任务该怎么做。
- 内容通常包括目标、步骤、判断标准、输入输出格式、注意事项和示例。
- 它解决的是“遇到这类任务，应该怎么做”。

作者强调 `Skill` 的价值在于：

- 把团队经验、领域知识、质量标准沉淀为文本。
- 不需要重新训练模型。
- 不需要每次都重新手写长 prompt。

但它也有边界：

- `Skill` 不是工具，不负责真正执行外部动作。
- `Skill` 更像方法论和工作规范。
- 当模型在某个领域已经很强时，过重的 Skill 反而可能限制灵活性。

### CLI vs MCP

slides 接着比较了两种接入能力的方式：

- `CLI` 更适合开发内环，快、透明、可组合、可调试。
- `MCP` 更适合标准化工具接入，具备可发现性、结构化输入输出和跨客户端复用能力。

一个偏“直接干活”，一个偏“能力协议化”。

## 4. 从想法到交付的实际 Workflow

slides 给出了一套作者自己的 AI 协作工作流：

1. 发现需求
2. 用 GPT 聊天写计划
3. 用 Stitch 绘制 demo 图
4. 用 Google AI Studio 输出原始代码
5. 用 Codex 初始化项目
6. 把计划书注入上下文
7. 根据实际效果不断迭代
8. 最终审核代码提交

这个 workflow 体现出一个很重要的思路：

- 前半段是“想法成型 + 快速出雏形”
- 后半段是“Codex 接手本地工程 + 人工把关”

也就是说，AI 的价值不只是生成代码，而是贯穿从构思、原型、脚手架到落地代码的整条链路。

## 5. Research、Prompt 与设计协作

slides 中段强调了三类关键能力：

### Research 双模式

作者把 Research 分成两种模式：

- `Context-free`：不带先入为主偏见，基于第一性原理调研。
- `Context-rich`：结合当前项目限制和已有结构，在真实上下文中落地。

这两种模式组合起来，才更容易得到高质量设计决策。

### Prompt 经验

作者总结了几条个人 prompt 原则：

- 基于第一性原理思考，不套模板。
- 从根源解决问题，不只做表层修补。
- 对自己的思路保持中立，让 AI 可以挑战既有方案。
- 先理解现有代码，再开始改动。
- 相信 AI 的基础能力，不要过早把路径写死。

本质上，这些原则都在提升 AI 的自主搜索和判断空间。

### 不只是 Vibe Coding

作者认为 AI 不只是写代码，还可以成为沟通和协作介质，尤其体现在：

- 设计交互协作：开发者可以先用 AI 快速出原型，再和设计师对齐。
- 会议讨论准备：把调研与洞察变成可交互的讨论材料。
- 辅助代码理解：复杂技术方案用可交互文档表达，比长篇文字更容易减少误解。

这里的核心思想是：AI 可以把“抽象讨论”转成“可见、可点、可操作”的对象。

## 6. 用 AI 做设计的风险与机会

slides 讨论了一个很现实的问题：AI 生成设计很容易出现“AI 味”。

作者对“AI 味”的定义不是“做得差”，而是：

- 平均化
- 缺少取舍
- 缺少个性

即使通过 Skill 可以降低这种平均化趋势，也会带来新的问题：如果大家都用同样的 Skill，Skill 本身会不会成为新的“平均设计模板”？

这部分的判断不是反对 AI 设计，而是提醒：AI 能加速出稿，但人仍然要负责审美判断和取舍。

## 7. 深度使用 AI 后带来的变化

后半部分从“工具”转向“习惯变化”。

### 信息获取方式在变

作者认为最明显的变化是：从“构造 query”转向“直接表达需求”。

以前：

- 需要想关键词
- 多次换 query
- 自己筛选搜索结果

现在：

- 直接描述问题和上下文
- 让 LLM + Web Search 一次性完成诊断和方案生成

这不是说单次搜索一定更快，而是“拿到答案的整体链路”更短。

### 默认不信任“快”

一个很有代表性的感受是：用了 AI 以后，反而更信任“慢”。

原因是：

- 太快容易让人怀疑 AI 是否跳过了思考。
- “慢”往往意味着它真的在做 plan、research、验证。

这其实是把对人的判断方式迁移到了对 AI 的判断：越复杂的问题，越不该期待三秒钟内得到可信答案。

## 8. AI 正在从 coding agent 走向通用 agent

slides 还强调了一个扩展观点：Coding Agent 不只是会写代码的工具，它实际上是一种“在数字环境里解决问题的方法论实现”。

因此它可以延伸到很多生活场景，例如：

- 整理相册
- 刷新简历
- 自动生成赛历或订阅
- 制作演讲 slides

这里的重点不是“AI 什么都能做”，而是：一旦问题可以被描述、拆解、执行、验证，Coding Agent 的工作方式就能迁移过去。

## 9. 结尾：AI 是协作对象，不只是代码生成器

最后的小游戏和 Q&A 部分让整个分享收束到一个更轻松的氛围里，但前面的主线其实已经很清楚：

- AI 编程工具已经从补全进入 Agent 阶段。
- 真正的关键不只是模型，而是 `Prompt / Context / Agent / Skill / Tools` 的协同设计。
- AI 的价值不只是写代码，而是帮助人完成研究、沟通、设计、原型和落地。
- 人的角色并没有消失，而是从“逐行执行者”转向“目标设定者、上下文提供者、质量把关者和最终决策者”。

## 一句话摘要

这份 slides 的核心观点可以概括为：

> AI 不只是更快的补全工具，而是一个需要被设计、被协作、被信任、也需要被约束的工作伙伴。
